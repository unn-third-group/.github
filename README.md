### Подробное руководство по использованию репозитория `.github` на GitHub

Репозиторий `.github` — это специальный репозиторий, который GitHub автоматически распознаёт для хранения **общих конфигураций и ресурсов**, применяемых ко всем репозиториям организации или аккаунта. Он упрощает стандартизацию процессов, документации и автоматизации, избавляя от необходимости дублировать настройки в каждом репозитории.

---

## 1. Основные функции и файлы

### 1.1. Шаблоны Issues и Pull Requests
**Для чего:** Стандартизировать создание задач и запросов на слияние.  
**Как использовать:**  
- Создайте директории `.github/ISSUE_TEMPLATE` и `.github/PULL_REQUEST_TEMPLATE`.  
- Добавьте файлы в формате `.md` (например, `bug_report.md` или `feature_request.md`).  

**Пример `bug_report.md`:**
```markdown
---
name: Bug Report
about: Сообщить об ошибке
title: "[BUG]"
labels: bug
---

**Описание ошибки**  
...

**Шаги для воспроизведения**  
1. ...
2. ...
```

GitHub будет автоматически подтягивать эти шаблоны при создании нового Issue или PR.

---

### 1.2. Workflow Templates для GitHub Actions
**Для чего:** Использовать общие CI/CD-сценарии во всех репозиториях.  
**Как использовать:**  
- Создайте директорию `.github/workflows-templates`.  
- Добавьте YAML-файлы (например, `ci.yml`).  

**Пример `ci.yml`:**
```yaml
name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Hello, GitHub Actions!"
```

**Примечание:** Пользователи могут выбирать эти шаблоны при добавлении нового workflow через интерфейс GitHub.

---

### 1.3. Файл `CODEOWNERS`
**Для чего:** Назначить ответственных за код.  
**Как использовать:**  
- Создайте файл `.github/CODEOWNERS` в корне репозитория.  
- Укажите правила для проверки изменений.  

**Пример:**
```
# Ответственные за весь репозиторий
* @org/team

# Ответственные за директорию src
/src/ @user1 @user2
```

---

### 1.4. Документация
- **CONTRIBUTING.md**: Руководство для контрибьюторов.  
- **SECURITY.md**: Политика безопасности (например, как сообщать об уязвимостях).  
- **SUPPORT.md**: Информация о поддержке.  

Эти файлы автоматически отображаются в интерфейсе GitHub при создании Issue или PR.

---

### 1.5. Настройки по умолчанию (`settings.yml`)
**Для чего:** Управлять параметрами репозиториев через GitHub API.  
**Как использовать:**  
- Создайте файл `.github/settings.yml`.  
- Укажите общие настройки, например:  
```yaml
repository:
  name: .github
  description: Общие настройки организации
  has_wiki: false
  has_projects: false
```

**Примечание:** Для применения этих настроек используйте инструменты вроде [probot/settings](https://github.com/probot/settings).

---

### 1.6. Funding (`FUNDING.yml`)
**Для чего:** Добавить кнопки для поддержки проекта (GitHub Sponsors, Patreon и др.).  
**Пример:**
```yaml
github: [user1, user2]
patreon: org-name
```

---

### 1.7. Шаблоны для GitHub Discussions
**Для чего:** Стандартизировать обсуждения.  
**Как использовать:**  
- Создайте директорию `.github/DISCUSSION_TEMPLATE`.  
- Добавьте шаблоны в формате `.md`.

---

### 1.8. Dependabot Config
**Для чего:** Автоматическое обновление зависимостей.  
**Пример файла `.github/dependabot.yml`:**
```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
```

## 2. Как создать репозиторий `.github`
1. Перейдите в организацию или аккаунт.  
2. Создайте новый репозиторий с именем `.github`.  
3. Убедитесь, что репозиторий **публичный**, если хотите, чтобы файлы (например, шаблоны Issues) были видны всем.  
4. Добавьте нужные файлы и директории (см. раздел 1).  

---

## 3. Лучшие практики
- **Структура репозитория:**  
  ```
  .github/
  ├── ISSUE_TEMPLATE/
  ├── WORKFLOWS-TEMPLATES/
  ├── CODEOWNERS
  ├── CONTRIBUTING.md
  ├── SECURITY.md
  └── FUNDING.yml
  ```
- **Тестирование:** Проверьте, как файлы работают в тестовом репозитории.  
- **Обновления:** Изменения в `.github` применяются только к новым репозиториям. Для существующих может потребоваться ручное обновление.

## 4. FAQ
**Вопрос:** Что если репозиторий переопределяют файлы из `.github`?  
**Ответ:** Локальные файлы в репозиториях имеют приоритет. Например, если в репозитории есть свой `CONTRIBUTING.md`, он будет использоваться вместо общего.

**Вопрос:** Можно ли использовать `.github` в личном аккаунте?  
**Ответ:** Да, но некоторые функции (например, настройки организации) доступны только для организаций.

**Вопрос:** Как добавить шаблоны для Pull Requests?  
**Ответ:** Создайте файл `.github/PULL_REQUEST_TEMPLATE.md` или поместите несколько шаблонов в поддиректорию `PULL_REQUEST_TEMPLATE`.

## Заключение
Репозиторий `.github` — это мощный инструмент для стандартизации процессов в вашей организации. Он позволяет централизованно управлять CI/CD, документацией, политиками и автоматизацией, экономя время и уменьшая ошибки. Начните с малого — добавьте шаблоны Issues и Workflows, а затем расширяйте конфигурацию по мере необходимости.
